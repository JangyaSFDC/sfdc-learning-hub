Understanding Apex Triggers: A Complete Guide for Beginners

SH
SFDC Hub Team
Published on March 20, 2026

What are Apex Triggers?

Apex triggers are pieces of code that execute before or after specific data manipulation language (DML) events occur in Salesforce. They allow you to perform custom actions when records are inserted, updated, deleted, merged, upserted, or undeleted.

Triggers are essential for implementing complex business logic that cannot be achieved through declarative tools like Flows or Validation Rules.

Trigger Syntax

The basic syntax for an Apex trigger includes the trigger keyword, the object it fires on, and the DML events it responds to:

trigger AccountTrigger on Account (before insert, before update, after insert, after update) {
    // Your trigger logic here
    if (Trigger.isBefore && Trigger.isInsert) {
        // Handle before insert logic
        for (Account acc : Trigger.new) {
            if (acc.Industry == null) {
                acc.Industry = 'Other';
            }
        }
    }
}

Trigger Context Variables

Salesforce provides several context variables that give you access to the records that caused the trigger to fire:

  • Trigger.new — Returns a list of the new versions of the sObject records
  • Trigger.old — Returns a list of the old versions of the sObject records
  • Trigger.newMap — A map of IDs to the new versions of the records
  • Trigger.oldMap — A map of IDs to the old versions of the records
  • Trigger.isBefore / Trigger.isAfter — Context booleans
  • Trigger.isInsert / Trigger.isUpdate / Trigger.isDelete — Operation type

Before vs After Triggers

Understanding when to use before vs after triggers is crucial for writing efficient code:

Before Triggers

Use before triggers when you need to validate or modify field values before they are saved to the database. Since the record hasn't been committed yet, you can modify fields directly on Trigger.new records without performing an additional DML operation.

After Triggers

Use after triggers when you need to access field values that are set by the system (such as the record's Id or LastModifiedDate), or when you need to make changes to other records. In after triggers, Trigger.new records are read-only.

Trigger Best Practices

  1. One trigger per object — Consolidate all logic into a single trigger per sObject to maintain control over execution order
  2. Logic-less triggers — Keep triggers thin by delegating business logic to handler classes
  3. Bulkify your code — Always write triggers that can handle collections of records, not just single records
  4. Avoid SOQL/DML in loops — Collect data in lists and perform operations outside of loops
  5. Use trigger frameworks — Implement a trigger framework for consistent pattern and recursion control

Trigger Handler Pattern

A common best practice is to use a trigger handler pattern. The trigger itself is minimal, and all logic lives in a separate handler class:

// Trigger
trigger AccountTrigger on Account (before insert, before update, after insert, after update) {
    AccountTriggerHandler handler = new AccountTriggerHandler();
    handler.run();
}

// Handler Class
public class AccountTriggerHandler {
    public void run() {
        if (Trigger.isBefore && Trigger.isInsert) {
            beforeInsert(Trigger.new);
        } else if (Trigger.isBefore && Trigger.isUpdate) {
            beforeUpdate(Trigger.new, Trigger.oldMap);
        } else if (Trigger.isAfter && Trigger.isInsert) {
            afterInsert(Trigger.new);
        }
    }

    private void beforeInsert(List<Account> newAccounts) {
        // Before insert logic
    }

    private void beforeUpdate(List<Account> newAccounts, Map<Id, Account> oldMap) {
        // Before update logic
    }

    private void afterInsert(List<Account> newAccounts) {
        // After insert logic
    }
}

Summary

Apex triggers are powerful tools in the Salesforce developer's toolkit. By following best practices like using trigger handler patterns, bulkifying your code, and keeping triggers logic-less, you can build robust and maintainable automation that scales well within the platform's governor limits.
