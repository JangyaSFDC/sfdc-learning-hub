Breaking Trigger Implementation Step by Step — Trigger Design the Smart Way

SH
SFDC Hub Team
Published on March 24, 2026

The Scenario

Whenever an Account's Phone field is modified, update the related Contact records:

  • Contact OtherPhone → set to the old Account phone value
  • Contact HomePhone → set to the new Account phone value

This is a classic interview scenario that tests your ability to think through trigger design methodically. Let's break it down step by step before writing a single line of code.

Thinking Before Coding — The Analysis

Before jumping into code, let's figure out 3 things:

  • Object: Account
  • Event: after update — because we need to do something to another object (Contact), which means we need an "after" event
  • DML Operation: update — the phone field is being modified, not inserted

We also need to compare old value to new value to detect if the phone actually changed. This is done using Trigger.oldMap and Trigger.new.

Step 1: Identify Accounts Where Phone Changed

First, we collect the IDs of all Accounts where the Phone field was actually modified. We compare the new value (Trigger.new) against the old value (Trigger.oldMap).

// Step 1: Store Account IDs where Phone changed
Set<Id> accountIds = new Set<Id>();

for (Account acc : Trigger.new) {
    // Compare new phone to old phone
    if (acc.Phone != Trigger.oldMap.get(acc.Id).Phone) {
        accountIds.add(acc.Id);
    }
}

Why a Set? Sets prevent duplicates and are perfect for collecting IDs to use in SOQL WHERE clauses.

Step 2: Fetch Related Contacts

Now we query all Contacts that belong to the Accounts whose phone changed. We do this in a single query outside the loop to avoid governor limits.

// Step 2: Fetch contacts related to those accounts
List<Contact> relatedContacts = [
    SELECT Id, AccountId, HomePhone, OtherPhone
    FROM Contact
    WHERE AccountId IN :accountIds
];

Key Best Practice: Never put SOQL inside a for-loop! Always query once using a collection of IDs.

Step 3: Update Contact Fields

Finally, we iterate over the fetched Contacts, look up their parent Account's old and new phone values, and update the Contact fields accordingly.

// Step 3: Iterate and update Contact phone fields
List<Contact> contactsToUpdate = new List<Contact>();

for (Contact con : relatedContacts) {
    // Get the new Account record from Trigger.new via a map
    Account newAcc = Trigger.newMap.get(con.AccountId);
    // Get the old Account record from Trigger.oldMap
    Account oldAcc = Trigger.oldMap.get(con.AccountId);

    // HomePhone = new Account phone value
    con.HomePhone = newAcc.Phone;
    // OtherPhone = old Account phone value
    con.OtherPhone = oldAcc.Phone;

    contactsToUpdate.add(con);
}

// Bulk update all contacts
if (!contactsToUpdate.isEmpty()) {
    update contactsToUpdate;
}

The Complete Trigger

Putting it all together into a clean, bulkified trigger:

trigger AccountPhoneTrigger on Account (after update) {

    // Step 1: Collect Account IDs where Phone changed
    Set<Id> accountIds = new Set<Id>();

    for (Account acc : Trigger.new) {
        if (acc.Phone != Trigger.oldMap.get(acc.Id).Phone) {
            accountIds.add(acc.Id);
        }
    }

    // Only proceed if at least one phone changed
    if (accountIds.isEmpty()) {
        return;
    }

    // Step 2: Query related Contacts (SOQL outside loop!)
    List<Contact> relatedContacts = [
        SELECT Id, AccountId, HomePhone, OtherPhone
        FROM Contact
        WHERE AccountId IN :accountIds
    ];

    // Step 3: Update Contact phone fields
    List<Contact> contactsToUpdate = new List<Contact>();

    for (Contact con : relatedContacts) {
        Account newAcc = Trigger.newMap.get(con.AccountId);
        Account oldAcc = Trigger.oldMap.get(con.AccountId);

        con.HomePhone = newAcc.Phone;    // New value
        con.OtherPhone = oldAcc.Phone;   // Old value

        contactsToUpdate.add(con);
    }

    if (!contactsToUpdate.isEmpty()) {
        update contactsToUpdate;
    }
}

How to Think About Any Trigger — The Framework

Use this mental checklist every time you design a trigger:

  1. Which Object? — What object fires the trigger? (Account)
  2. Before or After? — Are you updating the same record (before) or a different record (after)?
  3. Which DML? — Insert, Update, Delete, or Undelete?
  4. What changed? — Compare Trigger.new vs Trigger.oldMap to detect field changes
  5. Collect IDs first — Use a Set to gather IDs of relevant records
  6. Query once — Fetch related data in a single SOQL query outside loops
  7. Process & update — Build a list of records to update, then do a single DML

Key Takeaways

  • Use after update when you need to modify records on a different object
  • Trigger.oldMap gives you the previous field values — essential for detecting changes
  • Trigger.newMap gives you the current/new field values
  • Always bulkify — collect IDs in a Set, query once, DML once
  • Early return — exit early if no relevant changes detected (saves resources)
