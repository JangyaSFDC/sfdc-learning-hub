Remember Order of Execution Like a Pro

SH
SFDC Hub Team
Published on March 24, 2026

What is Order of Execution?

When a record is saved in Salesforce (Insert or Update), Salesforce doesn't just save it directly. It runs through a specific sequence of operations in a defined order. This is called the Order of Execution. Understanding this sequence is critical for both Admins and Developers — it's one of the most asked interview questions!

Use this mnemonic to remember: “Very Angry Admin Work Pretty Efficiently”

The Complete Order of Execution

1. Prepare the Record

Salesforce loads the original record from the database (for updates) or initializes a new record (for inserts) and prepares it for processing.

2. Very → System Validation Rules

System-level validations run first — these are built-in checks like required fields, field formats, and field-length validation. You can't customize these; Salesforce handles them automatically.

3. Before Triggers Execute

All before insert and before update triggers fire. This is where developers can modify field values on the record before it's saved, without needing an extra DML operation.

4. Custom Validation Rules

Your custom validation rules run next. If any rule fails, Salesforce throws an error and stops the save process.

5. Record Gets Saved Temporarily

The record is saved to the database temporarily (not committed yet). The record now has an Id if it was a new insert.

6. After Triggers Execute

All after insert and after update triggers fire. The record is read-only in after triggers. Use these to update related records or perform cross-object logic.

7. Angry → Assignment Rules

Assignment rules execute (e.g., Lead Assignment Rules, Case Assignment Rules) to automatically assign records to the right owner or queue.

8. Admin → Auto-Response Rules

Auto-response rules fire next (e.g., automatic email responses for Web-to-Case or Web-to-Lead submissions).

9. Work → Workflow Rules

Workflow rules evaluate and execute their actions. This is important:

  • If a workflow has a field update, it modifies the record
  • That field update causes before and after triggers to fire again!
  • However, Salesforce prevents infinite loops — workflow rules won't re-evaluate a second time

10. Pretty → Process Builder

Process Builder processes execute. They can also trigger:

  • Roll-up Summary field recalculations
  • Grandparent Roll-up Summary recalculations (cross-object summaries)

11. Efficiently → Escalation Rules

Escalation rules evaluate, along with:

  • Criteria-based sharing rules — re-evaluate to grant/revoke record access

12. Record Gets Saved to Database

The record is committed to the database. It's now permanently saved.

13. After Commit Logic

Post-commit operations execute, such as:

  • Sending email notifications
  • Executing outbound messages
  • Enqueuing async Apex (future methods, queueable)

The Mnemonic — Remember It Forever

Map each word to the execution step:

V - Very        → System Validation Rules
A - Angry       → Assignment Rules
A - Admin       → Auto-Response Rules
W - Work        → Workflow Rules (+ field update re-triggers!)
P - Pretty      → Process Builder (+ Roll-up Summaries)
E - Efficiently → Escalation Rules (+ Sharing Rules)

Full sequence at a glance:

1.  Prepare record for Insert/Update
2.  System Validation Rules
3.  Before Triggers
4.  Custom Validation Rules
5.  Record saved temporarily (gets Id)
6.  After Triggers
7.  Assignment Rules
8.  Auto-Response Rules
9.  Workflow Rules (field update → re-fires triggers)
10. Process Builder & Roll-up Summaries
11. Escalation Rules & Criteria-Based Sharing Rules
12. Record committed to Database
13. Post-commit: Emails, Outbound Messages, Async Apex

Real-World Example

Let's trace what happens with a practical scenario:

Scenario: An Opportunity has Amount = 100. A Validation Rule fires if Amount < 1000. A Workflow Rule checks if Amount > 1000 and updates Amount to 100.

  1. User sets Opportunity Amount = 100
  2. Validation Rule checks: Amount < 1000? → Yes! Error fires, record is not saved
  3. User fixes it, sets Amount = 1500
  4. Validation Rule checks: Amount < 1000? → No, passes
  5. Record is saved temporarily
  6. Workflow Rule checks: Amount > 1000? → Yes! Updates Amount to 100
  7. Field update causes triggers to re-fire
  8. But Validation Rules do NOT re-evaluate on workflow field updates
  9. Final saved Amount = 100

This is a classic gotcha — the workflow can set a value that would normally fail validation, because validation rules don't re-run after workflow field updates!

Key Takeaways

  • Before triggers run before validation rules can modify field values
  • Workflow field updates re-trigger before and after triggers (but not workflow rules again)
  • Validation rules don't re-evaluate after workflow field updates — this is a common interview trick question
  • Record is committed only after all automation completes
  • Emails are sent after commit — so if the transaction rolls back, no emails go out
  • Use “Very Angry Admin Work Pretty Efficiently” to ace your interviews!
